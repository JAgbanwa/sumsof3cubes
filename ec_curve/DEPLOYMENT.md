# ec19n — Charity Engine BOINC Deployment Guide

## What this project searches

Integer solutions `(n, x, y)` where `y² = x³ + 1296n²x² + 15552n³x + (46656n⁴ − 19n)`
and `y/(6n)` is also an integer.

**Mathematical result:** Only `n ∈ {1, −1, 19, −19}` can ever yield `y/(6n) ∈ ℤ`
(proved by reducing mod n: requires n | 19).

## Quick start (Charity Engine / live BOINC server)

### 1. Provision a server

Any Debian 11/12 or Ubuntu 22.04/24.04 VPS. 2 CPU, 4 GB RAM is enough for the
server itself; volunteers do all computation.

Register at [Charity Engine](https://www.charityengine.com/) and request a
BOINC project slot, or self-host using the setup script below.

### 2. Upload project files

```bash
scp -r ec_curve/ user@your-server:~/ec19n_src/
```

### 3. Run the setup script (as root on the server)

```bash
sudo bash ~/ec19n_src/setup_ec19n_boinc.sh --host ec19n.yourdomain.com
```

This installs BOINC server software, creates a MySQL project database,
compiles the worker, registers the app, writes BOINC config, and starts
all daemons including the queue submission daemon.

### 4. Initialise the WU queue DB

Already done automatically by the setup script.  To run manually:

```bash
python3 /home/boincadm/projects/ec19n/ec19n_boinc_queue.py init
```

### 5. The submission daemon starts automatically

The BOINC daemon config (`config.xml`) runs:

```
python3 ec19n_boinc_queue.py submit --project_dir /home/boincadm/projects/ec19n
```

It maintains up to **5 000 WUs in-flight** at all times, submitting 500 per
90-second cycle as volunteers consume them from the queue.

### 6. Monitor progress

```bash
python3 /home/boincadm/projects/ec19n/ec19n_boinc_queue.py status
```

Example output:

## Maintenance

### After server downtime

Any WUs sent to volunteers but not returned will stay as `submitted` in the
inflight table. After 48 hours they are automatically re-issued. To force
immediate re-queue:

```bash
python3 ec19n_boinc_queue.py reset_stuck --stuck_hours 24
```

### Extend the search range

The default range is `|x| ≤ 10¹⁷`. To extend further, re-run `init` with a
larger `--x_max` after editing `X_MAX` and `X_BLOCK` in `ec19n_boinc_queue.py`.

## Standalone testing (no BOINC server)

```bash
# Generate 200 WU files (50 per n-value)
python3 ec19n_boinc_queue.py export --limit 200

# Run one WU file manually
./ec19n_worker wu_queue/ec19n_n+1_x-0000100000000000000_*.txt result.txt ckpt.txt
cat result.txt   # empty = no solutions in this block, correct

# Run all exported WUs in parallel (8 workers)
python3 ec19n_local_search.py --workers 8
```

## Worker binary

Build optimised for the local machine:

```bash
gcc -O3 -march=native -std=c99 -o ec19n_worker ec19n_worker.c -lm
```

For BOINC client (with checkpoint API):

```bash
gcc -O3 -march=native -std=c99 -DBOINC -o ec19n_worker ec19n_worker.c \
    -lboinc_api -lboinc -lm
```

Performance (Apple M-series reference):
- n=±1:  ~11 M x/s  (1/12 pass-through from residue filter)
- n=±19: ~33 M x/s  (1/228 pass-through)
